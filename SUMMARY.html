Architecture and Design Choices

Overview

To achieve the lowest valid playout delay, we must bypass the round-trip time (RTT) associated with traditional retransmission models. The system requirements specify a strict 2.0x bandwidth overhead cap on raw frames[cite: 1, 9]. We designed a hybrid Forward Error Correction (FEC) and ARQ mechanism that utilizes ~1.55x of this bandwidth budget to provide zero-RTT packet recovery.

Sender Architecture

The sender groups outgoing frames into pairs. For every sequential pair (e.g., Frame 0 and Frame 1), it generates a custom 168-byte parity packet containing the XOR computation of both payloads. This ensures that if the hostile network drops one of the primary frames, the redundant parity data is already en route[cite: 1, 7].

Receiver Architecture

  • Instant FEC Recovery: When the receiver detects an incoming parity packet (identified by a high-bit flag in the sequence header), it checks its tracking arrays. If one of the base payloads is missing, it instantly performs the XOR calculation to restore the dropped payload and forwards it to the playout harness[cite: 4, 6]. This entirely eliminates network delay latency.
  • ARQ Fallback: To handle "burst losses" where consecutive packets and their parities are dropped simultaneously by the relay[cite: 7], the receiver maintains a continuous 50ms select() loop[cite: 6]. It actively scans for unresolved gaps and fires lightweight 4-byte NACKs to the sender for delayed retransmission[cite: 3].
